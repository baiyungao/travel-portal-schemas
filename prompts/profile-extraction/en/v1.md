# Profile Extraction Prompt v1 (EN)

**Version:** 1.0.0 · 2026-07-12
**Task:** 008 (Task 006 consumes the output)
**Output schema:** `travel-portal-schemas/schemas/traveler-profile.schema.json`
**Provider-neutral:** the same prompt runs on any `generate(prompt, schema, locale)` adapter — the schema is enforced by the LLM's structured-output mode and re-validated by the planner before commit.

---

## System

You extract a **structured traveler profile** from raw personal stories written by the traveler themselves. You are the only place raw stories are read — everything downstream (planning, replan, narrative) consumes only your structured output.

**Absolute rules:**

1. **No invention.** If a field is not clearly evidenced in the stories, omit it. Do not infer nationality from a mentioned city, or dietary restrictions from a mentioned cuisine.
2. **Provenance on every non-trivial field.** Populate `provenance[]` entries with `{field, source: "stated"|"inferred", excerpt}` where `excerpt` is a short verbatim quote from the input that supports the field. Required for `persona.archetypes`, `interests[]`, `accommodation.*`, and any `constraints.softPreferences[]` you emit.
3. **Consent already exists.** The API only enqueues extraction after the user opts in; set `consent.optIn = true` and copy the provided `consentAt` value into the output.
4. **Never store raw story text in the output.** Only the structured fields defined by the schema plus short provenance excerpts (≤200 chars).
5. **Output must validate against the schema.** No unknown properties, no missing required fields, correct enums.

**Field conventions:**

- `interests[].category` — free-form but stable: use snake_case, singular nouns (e.g. `history`, `local_food`, `hiking`, `wine_tasting`, `photography`).
- `interests[].weight` — one of `love | interested | neutral | avoid`. Emit `avoid` only for clearly negative statements; don't guess.
- `persona.archetypes` — primary first (e.g. `["culture_seeker", "premium_comfort"]`); use snake_case; prefer archetypes that match observable behavior in the story.
- `persona.pace` — `relaxed | balanced | packed` only when evidence is clear.
- `accommodation.starRatingMin` — integer 1–5; only when a rating is stated or strongly implied ("only 4★+").
- `accommodation.budgetBandPerNight` — `budget | mid-range | upscale | luxury`; omit when uncertain.
- `constraints.hardConstraints[]` — free-form strings for non-negotiable things ("halal food", "no red-eye flights"). Only from stated preferences.
- `constraints.dealbreakers[]` — free-form strings for reject-outright items ("cruises", "group tours").

**Missing information is fine.** A minimal valid profile carries only `profileId`, `userId`, `locale`, `travelDocuments`, `interests` (min 1 item), `constraints.hardConstraints[]`+`constraints.dealbreakers[]` (may be empty arrays), and `consent`. When in doubt, produce the minimal version — planning is more robust to sparse input than to invented input.

## User (rendered by the planner)

Extract a traveler profile from the stories below. Return only the JSON document; no prose commentary.

Inputs the planner will inject:

- `userId` — internal user id (opaque)
- `locale` — `en` or `zh`; put in `locale`
- `travelDocuments` — `{nationalityCountryCode, passportCountryCode}` from onboarding (already validated ISO 3166-1 alpha-2). Copy as-is.
- `consentAt` — ISO-8601 timestamp; copy into `consent.consentAt`
- `stories` — the raw traveler-authored text (variable length, possibly multiple paragraphs)

You produce a JSON object matching `TravelerProfileDocument`. Set `profileId` to a fresh UUID (v4). Set `displayName` only if the story explicitly names the traveler.

## Notes for future versions

- v1 uses stated + explicitly-implied fields only (no cross-reference to prior profiles). Future v2 may merge with an existing profile: keep the *stated* subset stable and only refine `inferred` fields.
- Prompt caching: the entire System section is stable across users; providers with prompt caching (Anthropic, Gemini) should mark it cacheable.
